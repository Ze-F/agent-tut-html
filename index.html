0 · 心法先行

为什么要学这个：你大概已经让 AI 帮你写过东西——问一句，它答一段，挺聪明。但这本笔记要带你去的地方不止于此：让 AI 真正帮你把一件事做完，比如把你一周的物理笔记整理成一个能发到网上的网页。要做到这一点，光会「问得巧」不够，你得先有一套心法：怎么跟它合作、怎么盯着它别跑偏。这一章不教任何命令，只把心法讲清楚——后面九章都是在反复练这套心法。心法比 prompt 技巧更重要，因为技巧会过时，心法不会。

大脑 vs 大脑＋手脚

先分清两个词，后面全靠它。

大语言模型（large language model / LLM）是「大脑」：你给它文字，它给你文字。它能想、能写、能解释，但它看不见也碰不到你电脑里的文件、打不开网页、跑不了任何程序。就像一个隔着电话给你出主意的聪明朋友——主意再好，手还是你自己动。

智能体（agent）是「大脑＋手脚」：同一个大脑，外加一套能真正动手的工具——读你的文件、改文件、运行命令。Anthropic 给 agent 的定义很朴素：让模型「在一个循环里自主地使用工具，根据环境的反馈一步步推进」（原文：LLMs ... using tools based on environmental feedback in a loop）。说人话——它不再只是回你一段话，而是动手做、看结果、再决定下一步，直到把事办成。

给模型这双「手」的机制，官方叫工具使用（tool use，也叫 function calling）：模型自己判断该用哪个工具，发出一次结构化的调用，由程序去执行，再把结果还给它。一句话记住区别：

🧠 心法回扣 · 一句话区别

LLM＝会思考的大脑；agent＝大脑＋一双能在你电脑里干活的手脚。这本笔记教的，就是怎么用好这双手脚、并且始终由你说了算。

为什么是「现在」才好用

agent 这个想法不新，但近一两年才真的顺手，靠三件事凑齐了：

  • 模型够聪明了。当前旗舰模型专门擅长复杂推理和「长链条」的自主任务（Anthropic 形容最强档 Claude 适合 complex reasoning, long-horizon agentic coding, and high-autonomy work）——能扛住一件事的好多个步骤而不乱。
  • 会用工具了。有了上面说的 tool use，模型能真正调用工具去动手；官方说「给 agent 工具是最有效的能力之一」，加上基础工具后在不少任务上提升巨大。
  • 记性够大了。现在的上下文窗口（context window）很大——大到能一次「读进」你整批笔记、甚至整个项目文件夹，它才谈得上跨多个文件帮你干活。

核心比喻：一个超聪明、但对你一无所知的实习生

🧠 心法回扣 · 把 agent 当成实习生

整本书你只要记住一个画面：你手下来了一个智商极高、但对你一无所知的实习生。

他聪明、肯干、动作快，可是——他不知道你是谁、在做什么、想要什么样的结果，也没见过你电脑里任何东西。所以你越是把背景交代清楚（你在干嘛、给他哪些材料、你心里「做好了」长什么样），他越能帮上忙；你越是甩一句「帮我弄一下」就走开，他越可能认认真真做出一个跑偏的东西。

带 agent 和带实习生是同一件事：讲清楚 → 让他做 → 你验收 → 给反馈让他改。

主线心法：四步闭环

这套「带实习生」的节奏，拆开就是贯穿全书的四步闭环。每次让 AI 干活，都在转这个圈：

① 交代 context
讲清背景、给齐材料
→
② 产出
让它动手做出东西
→
③ eval 验收
你来检查对不对
→
④ 修正
指出问题让它改

④ 之后带着新反馈回到 ① ，圈越转越准 ↻

eval（验收 / evaluation）就是「你这个产出到底行不行」的检查动作——这一步是你的核心职责，不能外包给 AI。agent 自己也靠环境的反馈推进（官方说它需要「来自环境的实地结果，比如工具调用结果或代码运行结果，来判断自己进展如何」），但「这东西是不是你真正想要的」，最终得你来拍板。

两个最常见的坑，和解药

⚠️ 坑一：被 AI 牵着走

AI 答得又快又像模像样，你很容易就一路点「好」「继续」，最后交出一个你自己都没真正看懂的东西。问题不在它做了什么，而在你跳过了 ③ 验收——方向盘不知不觉交了出去。

⚠️ 坑二：AI slop

AI slop（AI 糊弄货）指那种「看着挺像样、其实空洞没价值」的产物：废话连篇的长文、似是而非的解释、能跑但你不理解的代码。它通常是「① context 没交代清楚 ＋ ③ 没认真验收」一起酿成的。

🧠 心法回扣 · 同一副解药

两个坑，解药是同一个：你当那个清醒的验收者，把四步闭环转完整。交代清楚 context（少踩坑二），认真做 eval 和修正（少踩坑一）。AI 负责动手，你负责定义「什么叫做好」——这件事永远是你的。

三条主线，会一路陪你练

从下一章起，这本笔记会用三个真实例子反复带你练这套心法。它们会反复出现，越走越深：

  • ① 课堂笔记 → 网页（HTML）→ 发布到 GitHub Pages：把你的物理（Physics）笔记交给 AI 整理成一个网页，最后用 GitHub Pages 发到网上、拿到一个能分享的链接。这是贯穿全书的主线项目。
  • ② 第二大脑：用 Obsidian（一个把笔记存成本地文件的工具）配合 Claude，搭一个会越用越聪明的第二大脑（second brain），把学到的东西沉淀下来、随时调用。
  • ③ 机械工程兴趣拓展：顺着你对机械工程（mechanical engineering）的兴趣，让 AI 帮你查、帮你整理、帮你拓展，把「随便看看」变成真能积累的探索。

这本笔记主要用 Claude Code（Anthropic 的命令行 agent 工具，官方定位：an agentic coding tool that reads your codebase, edits files, runs commands，住在你的终端里）。市面上同类工具还有 Gemini CLI（Google）、Codex CLI（OpenAI）等——它们路子相通，留到第 8 章再触类旁通，现在你只要知道「不止一家」就够了。

小结

  • LLM＝大脑（只产文字）；agent＝大脑＋手脚（在循环里用工具动手做事）。
  • 把 agent 当成超聪明但对你一无所知的实习生：你交代得越清楚，它越靠谱。
  • 主线心法是四步闭环：交代 context → 产出 → eval 验收 → 修正，转完再回到起点。
  • 两个坑——被 AI 牵着走、AI slop——同一副解药：你当清醒的验收者，定义「什么叫做好」。
  • 三条主线（笔记→网页、第二大脑、机械兴趣）会一路陪你把心法练熟。

一句话回顾：AI 是你那个超聪明的实习生，心法就是「交代清楚 → 让它做 → 你验收 → 让它改」；技巧会过时，这套心法不会。

1 · 三层界面各管什么

为什么要学这个：你已经知道 LLM 是"大脑"、agent 是"大脑＋手脚"——但具体在哪里用哪个？打开 claude.ai 对话窗口就够了，还是得开命令行？或者还有什么中间选项？这一章把三层界面说清楚，给你一个真正能用的决策框架：用完即走就留在网页；想让 agent 碰你的本地文件、但还没准备好命令行，有 Claude Cowork 接着；要长期维护项目、要发布上线、要完全掌控，再毕业到 Claude Code。搞清楚这三层，你后面不会走弯路。

先尝一口甜头：在网页里整理一段笔记

先别管界面怎么选，直接动手试一次——用 claude.ai（Anthropic 官方的网页 / 桌面 App）整理你的 Physics 课堂笔记。不需要装任何东西，打开浏览器就能做。

🙌 跟我做 · 示例① 起点：整理一段 2026-week3 笔记

1. 在浏览器里打开 claude.ai，登录你的账号。

2. 开一个新对话，把下面这段话（换成你自己这周的笔记内容）贴进去：

我是高中物理学生，下面是我 2026-week3 课堂的一段乱糟糟笔记，
帮我整理成干净的中英双语 HTML 格式，要有标题、要点、公式用 <em> 标出。
笔记：[把你的笔记粘在这里]

3. 发送，看 Claude 产出整理好的 HTML。

4. 试着问它"把第一条要点再解释得通俗一点"，练一次修正（四步闭环的 ④）。

这就是主线示例①「课堂笔记 → 网页（HTML）→ 发布到 GitHub Pages」的第一脚——你已经拿到了 AI 整理的 HTML 片段。

感受到了吗？交代 context（给笔记）→ 产出（Claude 整理）→ eval 验收（你看一眼）→ 修正（再问一句），四步闭环转了一圈，一分钟内。

撞墙：结果留不下来，也碰不到文件

现在问题来了：那段整理好的 HTML 只存在这一个对话里。你关掉页面、下周再想用，得重新来过。更关键的是——它碰不到你电脑里的文件。你没法让 claude.ai 直接帮你把整理好的内容写进你桌面上的 2026-week3.md，更别说帮你建整个文件夹结构、或者一次处理十几周的笔记。

这不是 Bug，这就是 claude.ai 网页/App 的设计边界：它的产出存在对话里，不在你的硬盘里（能接收你上传的文件分析，但分析完结果只在对话里，不会回写到硬盘）。要让 agent 真正动手碰你的本地文件，需要往下走一层——这时候就出现了两条路：一条零命令行，一条完全掌控。

决策框架：三层进阶模型

不是谁比谁好，是场景不同、能力不同。下面这张表帮你一眼定位：

claude.ai 网页 / App		 Research（联网多步研究）（需付费档）；Artifacts（交互可视化 / 迷你页）；上传文件快速分析；轻量问答探索 读写你硬盘上的文件；跨会话持久维护项目 用完即走：临时查资料、头脑风暴、整理一段笔记、生成可分享的交互小页

⚠️ Cowork 的平台限制

Claude Cowork 目前只在 Claude 桌面 App（macOS / Windows） 里运行，不支持网页端或手机端。访问需要付费档（Pro、Max、Team、Enterprise）。（来源：Anthropic 官方支持文档，2026-06-23 核实。）

Cowork 和 Claude Code 是什么关系？

官方说得直接：Claude Cowork 用的是「和 Claude Code 同一套 agentic 架构（same agentic architecture that powers Claude Code）」，只是入口搬进了 Claude 桌面 App 的 GUI 界面，不需要打开 terminal。两者的区别在于控制深度——Cowork 给你"零 terminal 动文件"的入门；Claude Code 给你"跑命令、版本控制、自动发布"的完全掌控。Cowork 不取代 Claude Code，而是它的无 terminal 入门版。

换个比喻：Cowork 像自动挡，Claude Code 像手动挡。自动挡先把你开上路，手动挡让你完全控制车子——路是同一条，发动机也是同一颗。

本地可视化：第 2 / 3 层都能做

你可能想："网页端的 Artifacts（交互可视化产物）挺酷的，本地能不能做？"能——在第 2 层让 Cowork 帮你生成一个 HTML 可视化文件、直接存到你的文件夹；或者到了第 3 层，让 Claude Code 写 Python 脚本画图，跑完结果留在你机器上、纳入版本控制。本地可视化的好处是产物是你的文件，随时能改，不依赖任何在线服务。

🧠 一句话原则

用完即走的探索 → 网页 / App（第 1 层）；想动本地文件又不想碰命令行 → Claude Cowork（第 2 层）；长期维护 / 发布 / 完全掌控 → Claude Code ＋ 备份（第 3 层）。

记不住表没关系，记住这句话。每次启动 AI 前先问自己两个问题："这件事做完就完了，还是我以后还要改？要不要碰我硬盘上的文件？" 答案给了，层级也给了。

心法在三层下完全一致

🧠 心法回扣 · 换了界面，心法不换

无论你在 claude.ai 网页里整理笔记、在 Cowork 里整理本地文件夹、还是在 Claude Code 里批量处理十几周的 Physics 笔记——四步闭环不变：交代清楚 context → 让它产出 → 你来 eval 验收 → 给反馈修正。

界面是工具，心法才是本事。工具换了心法还在，你永远是那个清醒的验收者。

小结

  • 第 1 层 · claude.ai 网页 / App：用完即走，适合联网研究（Research）、交互可视化（Artifacts）、轻量一次性探索。碰不到你的本地文件。
  • 第 2 层 · Claude Cowork：零 terminal、不写代码，却能直接读写本地文件 / 文件夹；跑在 Claude 桌面 App 里，和 Claude Code 同一套 agentic 引擎；付费档可用，仅限桌面（macOS / Windows）。是非技术者让 agent 动本地文件的最丝滑入门。
  • 第 3 层 · Claude Code（CLI）：完全掌控——读写文件、跑命令、版本控制、批量自动化、发布到 GitHub Pages。主线示例①「课堂笔记 → 网页（HTML）→ 发布到 GitHub Pages」的完整路线，在这一层完成。
  • Cowork 不取代 Claude Code，是它的无 terminal 入门版；terminal 与 Claude Code 是完整教学目标——是你明确要学的，定位成毕业到更强、更可控的一层。
  • 心法在三层下一致：四步闭环在哪里都管用，你永远是那个说了算的验收者。

一句话回顾：用完即走找网页（第 1 层）；想动本地文件又怕命令行找 Cowork（第 2 层）；要长期维护 / 发布 / 完全掌控，毕业到 Claude Code（第 3 层）——下一章先去认识命令行。

2 · terminal 与文件系统

为什么要学这个：上一章说了，第 2 层 Claude Cowork 能让你零命令行就动本地文件——那为什么还要学 terminal？因为要真正毕业到第 3 层 Claude Code——让 agent 帮你跑命令、管版本、发布到 GitHub Pages、做可以长期维护的项目——你就必须认识 terminal。光这一点就值得学。更有用的是：当你在 Claude Code 里看着 agent 自己打命令、翻文件夹，你会真正看懂它在干什么、哪步可能出问题、该怎么纠偏。这一章不要求你变成命令行高手，只要你读完之后，能在 terminal 里找到自己在哪、能看文件、能建文件夹——再加上后面让 Claude Code 带你跑，就够用了。

命令行 vs 图形界面：一个类比

你平时用 Mac 管理文件，靠的是图形界面（Graphical User Interface / GUI）：双击文件夹、把文件拖来拖去、右键重命名。这就像在餐厅指着菜单说"我要这个"——直觉、方便，但菜单只给你有限的选项。

命令行（command line），或者叫终端（terminal），则像直接跟大厨说话："帮我把 A 文件夹里所有以 2026 开头的 .md 文件，全部复制到 B 文件夹，然后把名字改掉。" 更费脑力，但能做点菜单里根本没有的事，而且——可以让 agent 帮你打那些字。

对你来说，terminal 最大的价值不是你自己背一堆命令，而是：看懂 agent 在做什么，在它做错时你能叫停、纠正。

打开 Terminal.app

macOS 自带了一个 Terminal（终端）应用，就叫 Terminal.app，不需要安装任何东西。

🙌 跟我做 · 打开你的第一个 terminal 窗口

1. 点击 Dock 里的 Launchpad 图标（像火箭的那个）。

2. 在搜索栏输入 Terminal，然后点击出现的 Terminal 图标。

（另一种方法：Finder → Applications → Utilities → Terminal.app 双击打开。）

你会看到一个黑色或白色的小窗口，最后一行有个闪烁的光标，大概长这样：

Last login: Mon Jun 23 ...
你的用户名@你的电脑名 ~ %

那个 % 符号是命令提示符（prompt），它说："好了，等你输命令。" 你输入的所有命令都会在这里打，然后按 Return 运行。

较新的 macOS 默认用的 shell 叫 zsh（读作"zee shell"），这也是 Claude Code 运行的环境。提示符里看到 % 就是 zsh，看到 $ 是 bash——你的应该是 %。（来源：Apple 官方 Terminal 文档，2026-06-23 核实。）

路径：你在文件系统里的"地址"

在 Finder 里，你靠视觉看到文件夹套文件夹；在 terminal 里，每个位置都有一个文字地址，叫路径（path）。

路径有两种：

  • 绝对路径（absolute path）：从根目录 / 出发，写出完整地址。如 /Users/你的用户名/study/physics-notes/——就像写"新加坡市裕廊东 123 号"，永远不会搞错。
  • 相对路径（relative path）：从当前位置出发，只写剩余部分。如你已经在 /Users/你的用户名/，那 study/physics-notes/ 就够了——就像跟已经在同楼的人说"三楼左转"，比写完整地址省事。

三个特殊符号要记住：

  • ~（tilde 波浪号）：你的家目录（home directory）的快捷方式，等价于 /Users/你的用户名/。你的文件大多放在这里的子文件夹里。
  • .（一个点）：当前目录本身。
  • ..（两个点）：上一级目录（父目录）。用 cd .. 往上退一层。

第一批命令：文件浏览

下面逐个认识最常用的命令，每个都有一个小练习——请在你自己的 terminal 里跟着做。

🙌 跟我做 · pwd：我在哪里？

pwd（Print Working Directory）告诉你当前在文件系统里的哪个位置。

pwd

预期输出（你的用户名会不一样）：

/Users/你的用户名

刚打开 terminal，你通常就在家目录（~）。看到这个路径就对了。

🙌 跟我做 · ls：这里有什么？

ls（List）列出当前目录里的文件和文件夹。

ls

预期输出（你看到的名字会不一样）：

Desktop    Documents    Downloads    Movies    Music    Pictures

试试加 -l 选项，看更多细节（文件大小、修改时间）：

ls -l

输出会变成每行一个文件，有日期和大小。加 -la 还会显示以 . 开头的隐藏文件（如 .zshrc）。

🙌 跟我做 · cd：走过去

cd（Change Directory）切换到另一个目录。

cd Documents

没有任何输出就对了——terminal 的风格是"没问题就沉默"。然后用 pwd 确认你已经进去了：

pwd

预期输出：

/Users/你的用户名/Documents

往回退一层：

cd ..

回到家目录（随时可用）：

cd ~

🙌 跟我做 · mkdir：新建文件夹

mkdir（Make Directory）新建一个文件夹。我们来建主线示例①的工作文件夹：

mkdir -p ~/study/physics-notes/raw

（-p 让命令一次建好多层文件夹，而且就算某层已经存在也不会报错——所以这条命令重复跑也安全。）

验证是否建好：

ls ~/study

预期输出：

physics-notes

你刚建好了主线项目的工作目录 ~/study/physics-notes/——Claude Code 之后就在这里帮你整理 Physics 笔记。

🙌 跟我做 · cat：看文件内容

cat（Concatenate and print）把文件内容打印到 terminal 里。先建一个测试文件（不用担心这步的语法，先照做即可）：

echo "Newton's second law: F = ma" > ~/study/physics-notes/raw/2026-week3.md

（这里的 > 是"写入"符号——如果文件已存在会覆盖它；这只是个测试文件，覆盖没关系。）

然后用 cat 看它：

cat ~/study/physics-notes/raw/2026-week3.md

预期输出：

这就是你的主线笔记文件的起点——之后会交给 Claude Code 整理。

🙌 跟我做 · cp：复制文件

cp（Copy）复制文件到另一个位置。

cp ~/study/physics-notes/raw/2026-week3.md ~/study/physics-notes/raw/2026-week3-backup.md

验证：

ls ~/study/physics-notes/raw

预期输出：

复制整个文件夹要加 -r（recursive 递归）：cp -r 源文件夹 目标位置。

🙌 跟我做 · mv：移动或重命名

mv（Move）既可以移动文件，也可以重命名——在同一个文件夹里"移动到新名字"就是重命名。

mv ~/study/physics-notes/raw/2026-week3-backup.md ~/study/physics-notes/raw/2026-week3-bak.md

验证（文件名已变）：

ls ~/study/physics-notes/raw

预期输出：

2026-week3-bak.md    2026-week3.md

🙌 跟我做 · open：用 Finder 打开

open 是 macOS 专有命令，能在 Finder 里打开文件夹，或者用默认 App 打开文件。

open ~/study/physics-notes

预期结果：Finder 跳出来，打开到 physics-notes 文件夹——你能看到刚才建的 raw/ 子文件夹。

特别实用的用法：open .（在 Finder 里打开当前目录）——当你在 terminal 里工作、想用 Finder 看一眼同一个地方时，就用这个。

⚠️ 安全盒：rm 是永久删除，没有废纸篓

rm（Remove）是最危险的常用命令。它直接永久删除文件，不会移到废纸篓（Trash），删了就删了，除非你恰好有 Time Machine 备份。

Finder 里把文件拖到废纸篓还有机会找回来；rm 没有。

用 rm 之前，养成一个习惯：先 ls 看清当前目录，用 pwd 确认你在哪里，再运行 rm。

# 删单个文件（只是示意，别把你需要的文件删掉）
rm ~/study/physics-notes/raw/2026-week3-bak.md

尤其要避免加 -rf 组合选项（强制、递归删整个文件夹）。一旦对着错误的路径跑，整个文件夹瞬间消失，无法撤销。Claude Code 有时会用 rm——这正是为什么你需要在旁边读懂它在做什么，而不是闭眼说"好"。

Terminal.app 补充：不必是你唯一的 terminal

macOS 自带的 Terminal.app 够用，但很多开发者偏爱 iTerm2（免费，功能更多）或者 VS Code 内建的 terminal。选哪个不影响命令，命令都是一样的。入门阶段用 Terminal.app 最省事，之后换不换随你。

Homebrew：terminal 里的"App Store"

Homebrew 是 macOS 上最流行的包管理器（package manager），用来在 terminal 里安装各种命令行工具——比如 git（版本管理）、node（跑 JavaScript）、python 等。官方介绍它是 "The Missing Package Manager for macOS"。以后用 Claude Code 时，它可能让你先装一些工具；知道 Homebrew 是什么，你就不会慌。

官方安装命令（来源：brew.sh，2026-06-23 WebFetch 核实）：

/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"

⚠️ 安装 Homebrew 前先确认

从网上下载脚本直接运行，风险不低。请确认：① 你是从官方网站 brew.sh 复制的命令；② 你的网络可信。如果 Claude Code 之后提示你安装某个工具，它通常会帮你生成对应的 brew install ... 命令，你确认后再跑就好。

装好 Homebrew 之后，安装工具的命令格式是 brew install 工具名，例如 brew install git。

🧠 心法回扣 · 你不必记住所有命令

这章的命令你不需要全背。真正的工作流是：你大概知道有哪些命令、它们能做什么 → Claude Code 帮你打出具体命令 → 你读懂、确认、按 Enter。 你的角色是"懂行的验收者"，不是"人肉搜索引擎背命令"。遇到不认识的命令，随时可以问 Claude。

小结

  • Terminal.app 是 macOS 自带的命令行窗口；打开方法：Launchpad → 搜索 Terminal。默认 shell 是 zsh（提示符 %）。
  • 路径是文件在系统里的"地址"：绝对路径从 / 开始；相对路径从当前位置算。~ 是家目录，. 是当前目录，.. 是上一级。
  • 核心浏览命令：pwd（在哪里）、ls（看什么）、cd（去哪里）、cat（看文件内容）、open .（在 Finder 里打开）。
  • 文件操作：mkdir（建文件夹）、cp（复制）、mv（移动 / 重命名）。
  • rm 永久删除，没有废纸篓，跑前先 ls + pwd 确认。
  • Homebrew 是 macOS 的包管理器，用来安装命令行工具；安装命令从 brew.sh 官网复制。
  • 你不需要背命令——懂原理、看得懂 agent 在做什么，才是重点。

一句话回顾：terminal 是你和 Claude Code 共同工作的地盘——学会"看懂它在哪、看懂它在做什么"，比背命令更重要；下一章直接开进 Claude Code。

3 · 上手 Claude Code

为什么要学这个：上一章你已经认识了 terminal、建好了 ~/study/physics-notes/，还在 raw/2026-week3.md 里写下了第一行 Physics 笔记。上上章（第 1 层 claude.ai 网页）帮你探过一次这份笔记——那是「问一句、答一段」，窗口关了什么都不剩。第 2 层 Claude Cowork 能让你零命令行就动本地文件，已经是一大步。但要把笔记变成一个真正能发布到 GitHub Pages 的网页、能用 git 管版本、能批量自动化处理，就需要毕业到第 3 层 · Claude Code：在 terminal 里运行、完全掌控你的文件系统、可编程、可维护、可发布上线。这一章就带你把它装好、登录、跑起来。

harness 是什么

Claude Code 官方文档有一句定义值得记住：

"Claude Code serves as the agentic harness around Claude: it provides the tools, context management, and execution environment that turn a language model into a capable coding agent."

Agentic harness（工作台 / 套具）——把模型想成一颗大脑，harness 就是把这颗大脑接上工具、让它真正能在你电脑里动手的整套框架。没有 harness，模型只能返回文字；有了 harness，它就能读文件、改文件、运行命令、看输出、再决定下一步。

一句话点清两者的区别：Cowork 和 Claude Code 用的是同一套 agentic 引擎——Cowork 住在桌面 App 里、零 terminal、鼠标点点就能动你的本地文件；Claude Code 住在 terminal 里、完全掌控、能跑任何命令。同一个大脑，工作台不同，能做的事就不同。

Claude Code 的 agentic loop：读→想→动手→再读

你下达一个任务之后，Claude Code 不是一次性甩给你答案，而是进入一个循环——官方叫它三阶段（gather context → take action → verify results），映射到你能感受到的节奏就是：

① 读
gather context
读你的文件和问题
→
② 想
（内部推理）
决定下一步怎么做
→
③ 动手
take action
改文件/跑命令
→
④ 再读
verify results
看结果、判断够不够

④ 之后带着新结果回到 ① ，圈越转越近目标 ↻　任务完成才停下来

注意：这个循环是 Claude Code 自己在跑的——但你在 ④ 阶段要和它一起做「verify」：看它改了什么、判断对不对，必要时叫停或纠正。agent 的循环和你的四步闭环（Ch0 心法）在这里合体。

主线示例①第一段：让 Claude Code 整理你的 Physics 笔记

前面几章你在 Ch2 建好了 ~/study/physics-notes/raw/2026-week3.md，里面有一行笔记起点。现在让 Claude Code 来读它、整理它——这是主线示例①的第一步（后面 Ch5 会把整理结果转成 HTML，然后发到 GitHub Pages）。

在 ~/study/physics-notes 目录里运行 claude 启动对话，然后把下面这段作为你的第一条指令（注意：把 context 交代清楚，这是 Ch0 心法）：

我在学 Physics（O-level 水平），raw/2026-week3.md 是我本周的课堂笔记，可能有些乱。请帮我读一下这个文件，然后把内容整理成条理清晰的 Markdown 格式——保留原来的公式和知识点，但让结构更好读。整理好后保存回同一个文件。

Claude Code 会依次经历那四步 loop：读（打开 raw/2026-week3.md）→ 想（决定怎么整理）→ 动手（改写文件内容）→ 再读（确认写入成功）。

它动手前通常会告诉你它打算做什么、并请求你确认（这是 Claude Code 的默认安全行为）。你看一下计划、觉得没问题就确认，觉得哪里不对就在确认前说出来。

它做完之后，你用 cat raw/2026-week3.md 或者直接在 Finder 里打开，验收整理结果对不对——这就是你的 eval 步骤。如果结构还不够好，继续在对话里说："第二节可以再细化一下"，它会接着改。

🧠 心法回扣 · 你负责 context 和 eval

刚才那条指令之所以好用，因为它交代了三件事：你是谁（Physics 学生）、材料在哪（raw/2026-week3.md）、你想要什么（整理成条理清晰的 Markdown）。这就是 Ch0 说的「交代清楚 context」。

Claude Code 在 agentic loop 里自己做「verify results」；但「这个结果是不是你真正想要的」——那个 eval，是你的事，不能外包给它。它整理好了，你开文件看一眼，才算真正走完一圈。

四步闭环：你给 context → 它动手 → 你 eval → 你给反馈让它改。

4 · 把心法用熟

为什么要学这个：到这里，你已经能让 Claude Code 在 terminal 里帮你动手了。但「能动手」和「帮你做对」是两回事。Ch0 给了你心法的骨架——四步闭环、两个坑（被 AI 牵着走、AI slop）；这一章把骨架填成肌肉：具体怎么把高质量 context 喂进去，又具体怎么做一次真正的 eval。这是全书的「魂」。把这两件事做扎实，方向盘就一直在你手里——AI 跑得再快，也跑不偏。这一章稍微长一点，因为它最重要。

先把闭环摆回眼前

Ch0 的四步闭环——交代 context → 产出 → eval 验收 → 修正——本质上只分两半：① 喂进去什么（context，输入半边），和 ③ 你怎么验收（eval，输出半边）。中间的「产出」是 AI 的事，「修正」是把 eval 的结论再喂回去。所以你真正要练熟的，就是这两件事。这一章前半教你喂高质量 context，后半教你做真正的 eval。下面所有的命令、模式、技巧，都只是为这两件事服务的工具——别把它们当功能清单背，要当成「怎么喂得更清楚」或「怎么验得更狠」来理解。

前半 · 怎么喂高质量 context

回到实习生的比喻：一个对你一无所知的实习生，你每天早上都得重新跟他自我介绍一遍——这太累了。如果能把「我是谁、在做什么、什么叫做好」写成一张纸贴在他桌上，他每次上工先看一眼，就省了无数次重复交代。Claude Code 里那张纸，叫 CLAUDE.md。

CLAUDE.md：写给 agent 的「项目说明书」

CLAUDE.md（项目记忆文件）是一个用 Markdown 写的纯文本文件，放在你的项目文件夹里。官方说得很直白：「你用纯文本写好它，Claude 在每次会话开始时读它」（原文：You write these files in plain text; Claude reads them at the start of every session.）。也就是说，你不用每次开新对话都重新交代背景——把背景写进 CLAUDE.md，它自动读。

⚠️ 关键认知：它是「建议」，不是「铁律」

很多人误以为 CLAUDE.md 是一份强制配置——写了「不许做 X」它就一定不做。不是这样。官方明确说：Claude 把它当成 context（上下文），而不是强制配置（Claude treats them as context, not enforced configuration.）。它会尽力遵守，但不保证 100%。说人话：CLAUDE.md 是你给实习生的「背景介绍 + 工作守则」，不是上了锁的保险箱。真要拦死某个危险操作，得靠权限规则（后半会讲），不是靠在这里写一句话。

怎么生成和管理它，有两个命令：

  • /init——让 Claude 自动分析当前项目，帮你生成一份 CLAUDE.md 草稿。第一次进一个新项目时很省事，生成完你再按自己的需要改。
  • /memory——查看和编辑当前会话加载了哪些 CLAUDE.md。想确认「它到底读到了什么背景」，用这个。

还有一条官方建议要记住：每个 CLAUDE.md 尽量控制在 200 行以内。太长了它反而抓不住重点、遵守率下降——就像守则写成一本书，实习生根本读不完。简洁、只写真正重要的规则，效果最好。

🙌 跟我做 · 给 Physics 项目写一份 CLAUDE.md

Ch3 你已经有了 ~/study/physics-notes/ 这个 workspace。现在给它配一张「实习生守则」，让以后每次对话它都先认识你。

第一步：进项目目录，启动 Claude Code

cd ~/study/physics-notes
claude

第二步：让它先帮你起个草稿（在对话里输入）：

/init

它会扫一眼这个文件夹，生成一份 CLAUDE.md 草稿。

第三步：把草稿改成真正描述「你」的内容。你可以直接让它改，比如这样说：

请把 CLAUDE.md 改写成下面这些内容（保持简短）：
- 我是谁：高中生，在新加坡读书，明年考 O-level，正在学 Physics。
- 这个项目是干嘛的：整理我每周的 Physics 课堂笔记，原始笔记在 raw/，整理后要能转成 HTML 发到网上。
- 我心里「整理得好」是什么样：结构清晰、保留所有公式和定义、中英术语都标出来、宁可啰嗦也不要漏知识点；解释要让一个刚学这章的人也能看懂。
- 永远不要做：不要自己删掉或简化我原始笔记里的内容；不确定的地方先问我，别猜。

第四步：用 /memory 确认它读进去了，看看加载的 CLAUDE.md 是不是你刚写的那份。

从此以后，你在这个文件夹里开任何对话，都不用再重新解释「我是谁、什么叫整理得好」——它开场就知道。这就是「把高质量 context 一次喂好、长期复用」。

建 workspace，也是在喂 context

别小看「把文件放进一个项目文件夹」这件事。Ch3 你把笔记放进 ~/study/physics-notes/，本身就是在给实习生整理工位——所有材料归在一处，他一进来就看得到全貌，而不是散落在桌面各处让他到处找。一个干净、命名清楚的 workspace，加上一份 CLAUDE.md，就是「高质量 context」的物理形态。

临时 context：把你的现状、诉求当场讲清楚

CLAUDE.md 装的是「长期不变」的背景（你是谁、什么叫做好）。但每次具体任务还有「这一次」的背景——这部分在对话里当场说。一条好指令通常交代三件事，和 Ch3 那条整理笔记的指令一模一样：我现在的状况（现状）＋我是谁（profile）＋我这次要什么（诉求）。

举个贴你生活的例子。明年 O-level，你想让 AI 帮你把这周的电学笔记整理成复习用的版本，好的说法是：

我明年考 O-level，这周学的是电路（current electricity），raw/2026-week3.md
是我的课堂笔记。帮我整理成一份「考前复习版」：每个公式后面补一句它什么时候用，
每个定义配一个最简单的例子。我容易混 series 和 parallel，这部分讲细一点。

对比一句「帮我整理一下笔记」——前者把现状（学了什么）、profile（O-level、容易混哪里）、诉求（复习版、补例子）全交代了，实习生才知道往哪使劲。context 喂得越具体，产出越不需要返工。这是从源头上少踩「AI slop」那个坑。

后半 · 怎么做真正的 eval

context 喂得再好，AI 也会出错——它毕竟是个动作飞快但不了解你全部心思的实习生。所以闭环的第三步 eval（验收），是你的核心职责，没法外包。Ch0 说过、Ch3 说过，这里要把它讲成可操作的动作。eval 的反面就是 Ch0 那个最大的坑——一路点「好」「继续」，把方向盘交出去。下面这些工具，全是帮你不交出方向盘的。

核心动作：看 diff，别盲目接受

AI 改完文件，第一件事不是说「谢谢」，是看它到底改了什么。命令是：

/diff

diff（差异对比）会把所有还没提交的改动逐行列出来：绿色是新增、红色是删掉的。你扫一遍，问自己三个问题：它改的是我想要的地方吗？它有没有顺手删了我不想删的东西？这些改动我看得懂吗？看不懂的，就追问它「这一段为什么这么改」——看懂了再算数。这一步就是把 Ch0「你自己都没真正看懂的东西」挡在门外。

把验收结论喂回去——这就是「修正」

看完 diff，你心里有了判断（这就是 eval），下一步是把判断说出来，让它改（这就是闭环第四步「修正」）。别只会说「不对」，要说清哪里不对、你想要什么：

第二节公式列对了，但你把我笔记里『右手定则』那段整段删了——
那个要保留。另外例子太难，换成更基础的。

它改完，你再 /diff 看一遍。看 diff → 给反馈 → 再看 diff，这就是四步闭环在 terminal 里转起来的真实样子。圈转得越勤，产出越贴你心意。

让权限提示帮你 eval：每次确认都是一个检查点

你大概注意到了：Claude Code 动手改文件前，常常会停下来问你一句「我打算这么做，可以吗？」——这不是麻烦，这是免费的 eval 检查点。它把「我准备做什么」摊在你面前，等你拍板。这个「停下来问不问、问哪些」的行为，由权限模式（permission mode）控制。Claude Code 一共有 6 个模式，但日常你只要先认得三个：

模式 它会自动放行什么 什么时候用
default
（默认）		 只有「读」操作自动放行；要改文件、跑命令都得问你 入门、做重要 / 敏感的事——你想盯紧每一步时
acceptEdits
（自动接受编辑）		 读 + 改文件 + 常见文件操作（建文件夹、改名等）都自动放行 你已经信任它在做的事、想少被打断地连续迭代时
plan
（计划模式）		 只读、只探索——完全不碰你的文件，只列出一份计划给你看 大改动动手前，先让它探索 + 出方案，你审完计划再放行

会话里按 Shift + Tab 就能在这三个模式之间循环切换，左下角会显示当前在哪个模式。这三个模式其实是一根「eval 严格程度」的滑杆：default 最严（步步问你）、acceptEdits 松一些（信任它动手、你事后用 /diff 补验收）、plan 则是「动手前先验收一次计划」。选哪个，取决于这件事你想盯多紧。

plan 模式：在它动手之前就 eval

plan 模式值得单独说，因为它把 eval 提前到了「动手之前」。官方定义：计划模式让 Claude 只研究、只提方案，但「不改你的源文件」（原文：but does not edit your source.）。它会读文件、探索，最后给你一份「我打算怎么做」的计划——你审这份计划，觉得行才放它去做。进入方式有两种：按 Shift + Tab 循环到 plan，或者直接输入命令并带上任务说明：

/plan 帮我把这学期所有 raw/ 里的笔记整理成统一格式

计划出来后，你可以批准它去执行、也可以说「这里改一下」让它重新规划。对「一次要动很多文件」的大任务，先 plan 一下，是性价比最高的 eval——在它动手之前就把方向纠正了，比事后看着一堆改动收拾要省事得多。

⚠️ 那几个「省心但危险」的模式，别乱开

除了上面三个，还有几个需要特殊开启的模式，越往后越「省心」、也越容易让你把方向盘整个交出去——这正是 Ch0 第一个坑：

  • auto（自动模式）——几乎什么都不问你、自己往下跑（有后台安全检查）。适合你确实信任的长任务，但代价是你看到的 eval 检查点变少了。
  • dontAsk——只对你事先在规则里允许过的操作不再追问，其余照样拦；多用于自动化 / 流水线场景，新手用不上。（连同上面 auto、下面 bypassPermissions，就是前面说的「6 个模式」里日常用不到的那 3 个。）
  • bypassPermissions（跳过所有检查）——顾名思义，所有权限确认全部跳过，连受保护的路径都直接动。官方建议它只在隔离的容器 / 虚拟机里用。在你自己的 Mac 上、对着你真实的文件，不要开它。一时图省事跳过所有确认，等于把那个「不了解你全部心思的实习生」单独留在你家里、还把所有抽屉钥匙给了他。

新手就老老实实在 default / acceptEdits / plan 之间转。「少被打断」永远不该用「不再验收」换。

顺带一提，受保护的路径（比如 .git、你的 ~/.zshrc 这类）即使在 acceptEdits 下要写也会照样问你——只有 bypassPermissions 会把这层保护也关掉。想细看 / 管理具体的权限规则，用 /permissions。

拆长任务：context 清楚 + eval 容易，一举两得

有个习惯能同时改善 context 和 eval：别把一个庞大模糊的任务一口气甩给它。「帮我把这学期笔记全弄好」太大太糊——它不知道从哪下手（context 不清），你也没法验收（eval 无从下口）。拆成「先整理 week3 这一篇」「再统一所有标题格式」「最后转成 HTML」，每一小步都好交代、好验收。这也正是 plan 模式帮你做的事：把大任务拆成清单，你逐项过目。

管好 context 这块「黑板」：/clear vs /compact vs /context

AI 的 context window（上下文窗口）像一块有限的黑板：聊得越久，黑板写得越满，早期的内容会被挤掉、它也容易「忘事」或被无关的旧对话干扰。管好这块黑板，本质上是保证它手上的 context 始终干净、相关——这又回到了「喂高质量 context」。三个命令各管一摊：

命令 它做什么 什么时候用
/context 显示黑板现在用了多少（像个进度条） 想知道「快满了吗」，先看一眼
/compact 把当前对话压缩成摘要、腾出空间，继续同一个任务 同一个长任务还没做完，但黑板快满了
/clear 擦掉黑板、开一段全新对话（旧对话有保存，可恢复） 完全换一件事，不需要之前的上下文了

一句话分清最常混的两个：同一个任务、想接着干，用 /compact；换个新任务、之前的不要了，用 /clear。

🙌 跟我做 · 三个 eval 习惯，今天就练

下次让 Claude Code 帮你改 Physics 笔记时，刻意做这三下：

1. 它改完，先打 /diff 看改了什么，别急着说「好」。

2. 看到不对的地方，具体说出哪里不对、你要什么，让它改——然后再 /diff 一遍。

3. 这件事彻底做完、要换下一件事时，打 /clear 给黑板擦干净再开始。

三下做满，你就完整转了一圈四步闭环，而且方向盘从头到尾在你手上。

⚠️ 注意 · 别让「确认」变成肌肉记忆

最隐蔽的坑不是某个危险命令，是你对着权限提示无脑按回车——提示框还在，你已经习惯性放行了。那一刻 eval 名存实亡，和开 bypassPermissions 没区别。每个确认框，都值得你真的读一眼它要做什么再决定。慢半秒，方向盘就还在你手里。

🧠 心法回扣 · 一切都挂在四步闭环上

这一章看着工具很多，其实只有两件事：

  • 喂 context（闭环第①步）：CLAUDE.md 装长期背景、对话里讲清这次的现状/profile/诉求、建好 workspace、拆小任务、/clear 与 /compact 让黑板保持干净——全是「让它更懂你要什么」。
  • 做 eval（闭环第③步）：/diff 看改动、权限提示当检查点、plan 模式提前审计划、看完把结论喂回去（第④步修正）——全是「让你始终说了算」。

AI 负责产出（第②步）；context 和 eval 这两半，永远是你的。工具会更新换代，这个分工不会变。

小结

  • 四步闭环 = 喂 context（输入半）+ 做 eval（输出半），中间产出归 AI、修正是把 eval 结论喂回去。
  • 喂 context：CLAUDE.md 写长期背景（/init 起草、/memory 查看、控制在 200 行内），它是建议不是铁律；对话里讲清现状/profile/诉求；建干净 workspace；拆小任务。
  • 做 eval：/diff 看改动、不盲目接受、把结论喂回去再看一遍；权限提示是免费检查点。
  • 权限模式当 eval 滑杆：新手在 default / acceptEdits / plan 之间用 Shift+Tab 切换；plan 让你在它动手前就审计划；auto / bypassPermissions 省心但危险，真实文件上别开 bypassPermissions。
  • 管黑板：同一任务用 /compact、换任务用 /clear、/context 看占用。

一句话回顾：把心法用熟，就是把两件事做扎实——用 CLAUDE.md 和清楚的指令喂高质量 context，用 /diff 和权限模式做真正的 eval；中间 AI 动手，方向盘始终在你手上。

5 · 让产出沉淀

为什么要学这个：Ch3 你让 Claude Code 把 raw/2026-week3.md 整理好了，Ch4 你学会了怎么把心法用扎实。但整理好的笔记还只是你自己电脑里的一个文件——关了 terminal、换了 Mac，什么都带不走；更别说分享给同学复习。这一章，我们把产出「沉淀」成真正耐用、随时随地可用的形态：先把 Markdown 变成一个 HTML 网页，再用 GitHub Pages 发到网上，得到一条任何设备都能打开的链接。这是主线示例①的最后一步——从「课堂笔记」到「网页（HTML）」到「发布到 GitHub Pages」，全部跑通。

Markdown 速览

Markdown（标记语言）是一种用纯文本符号来表示格式的写法。你写的时候只是普通文字 + 几个符号，渲染出来就有标题、加粗、列表。Ch3 整理好的笔记就是 Markdown 格式——现在先花两分钟把它认清楚，后面理解「为什么 AI 喜欢它」就更自然了。

核心语法只有几条（来自 GitHub 官方 Markdown 文档）：

# 一级标题（最大）
## 二级标题
### 三级标题

**这是加粗文字**

- 无序列表项 A
- 无序列表项 B

1. 有序列表第一项
2. 有序列表第二项

[链接文字](https://pages.github.com/)

Use `git status` 行内代码就这样写

🙌 跟我做 · 写一小段 Markdown

打开 terminal，用 Claude Code 在你的 Physics 笔记 workspace 里随手写一小段 Markdown：

cd ~/study/physics-notes

进入对话后，输入：

帮我在 raw/ 文件夹里新建一个 sample.md，内容包含：
一个二级标题「测试 Markdown」、一句加粗的定义、一个三项无序列表、
以及一行行内代码示例。

它生成完，你用 Finder 打开 raw/sample.md 看一眼，确认五种语法都在里面。这个文件只是练手，不影响正式笔记。

为什么 Markdown / HTML 是 AI 时代的通用格式

Word、Keynote、Pages 这些工具每个人都用过——但在 AI 工作流里，它们其实是「麻烦的格式」。下面几点说清楚为什么 Markdown 和 HTML 会成为通用选择（以下为基于事实的合理陈述，不夸大）：

  • 纯文本，AI 可直接读写。Markdown 文件是 .md 文本文件，Claude Code 无需插件即可生成、编辑、搜索。Word / PDF 要转换才能处理。
  • 版本控制（version control）友好，可以 diff。纯文本文件在 git（版本控制系统）里可以逐行追踪改动；Word、Keynote 这类二进制文件做不到这点。
  • Markdown → HTML 的管道成熟。把 Markdown 转成 HTML 的工具（如 Pandoc、marked.js 等）已经标准化，Claude Code 可以直接输出 HTML 文件，浏览器打开不需要本地服务器。
  • GitHub Pages 原生支持。GitHub 可以把 .md 或 .html 文件直接发布成网站，学习成本极低，是这一章要用到的路径。

一句话总结：Markdown 让 AI 写得顺、人看得清、git 管得住；HTML 让浏览器渲染出来、GitHub Pages 发出去。两个格式接在一起，就是「能 AI 处理 + 能上网发布」的最短路径。

把产出推上 GitHub

有了 site/index.html，下一步是把它推到 GitHub（代码托管平台）上，然后用 GitHub Pages（静态网站托管服务）发布成一个真实的网址。这一段是整章最容易卡住的地方，所以我们一步一步来，把每个可能的坑都踩平。

⚠️ 注意 · 先把 GitHub 账号和 git 认证一次配好

推代码到 GitHub 需要两样东西：本地有 git、并且 GitHub 知道「这台 Mac 是你的」（即认证 / authentication）。一次配好，以后每次 push 都不用重复。

第一步：检查 git 是否已安装

在 terminal 输入：

git --version

如果输出一个版本号（如 git version 2.x.x），说明 git 已经就绪，跳到第二步。

如果 macOS 弹出一个提示框，问你要不要安装「Command Line Tools」，点「安装」就行——整个过程约 5 分钟，不需要额外配置。也可以手动触发这个安装提示：

xcode-select --install

（注：macOS 的 /usr/bin/git 本身是一个占位符，实际指向 Xcode Command Line Tools。在全新 Mac 上第一次运行 git 时，macOS 通常会自动弹框提示安装。Apple 官方文档没有在本章研究中被直接验证，上述行为基于 Apple Developer Forums 记录——建议直接运行命令、看 macOS 的实际提示为准。）

第二步：注册 GitHub 账号（如果还没有）

去 github.com 注册一个免费账号。用户名会出现在你未来的 Pages 网址里（https://<用户名>.github.io/physics-notes/），选一个你满意的。

第三步：安装并登录 GitHub CLI（gh）

推荐这条路径，原因是：gh auth login 选 HTTPS 并回答 Yes 之后，会自动帮你存好 git 的认证信息，后续 git push 不再需要输入密码或配置 SSH key。对比网页拖拽上传（每次繁琐、不支持 push）和 GitHub Desktop（依赖图形界面、命令行技能无法迁移），gh 是新手上 terminal 最省事的认证路径。

如果还没装 Homebrew（macOS 的包管理器），先装它：

/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"

然后安装 GitHub CLI：

brew install gh

安装完成后，运行登录命令：

gh auth login

它会问你几个问题，按下面选：

  • 选择平台 → GitHub.com
  • 选择协议 → HTTPS（推荐新手，不需要 SSH key）
  • 「Authenticate Git with your GitHub credentials?」→ Yes
  • 选择认证方式 → Login with a web browser
  • 复制屏幕上出现的一次性代码，浏览器会自动打开，粘贴代码，点击授权

完成后，git push 就再也不需要输入密码了。这一步只需要做一次。

🧠 心法回扣 · 「沉淀」是把产出接上世界

Ch0 的心法是四步闭环：交代 context → 产出 → eval → 修正。这一章把「产出」这一步做到了底——不是产出一个临时文件，而是产出一个能被世界访问的页面。

Markdown 让 AI 写得顺；HTML 让浏览器渲染得出来；git 让你改动有记录；GitHub Pages 让任何人都能访问。这四层叠在一起，你的笔记就从「本地草稿」变成了「可维护的作品」。

同样的道理适用于任何产出：AI 帮你生成的东西，如果只活在对话窗口里，窗口一关就消失了；只有沉淀进文件、进版本控制、进公开发布，产出才真正属于你。

小结

  • Markdown：纯文本 + 符号 = 有格式的文档；AI 可直接读写、git 可 diff、Markdown → HTML 管道成熟。
  • 让 Claude Code 转 HTML：在 ~/study/physics-notes 目录里说清楚需求，它读 raw/2026-week3.md、输出 site/index.html；你 eval 验收，觉得有问题继续在对话里改。
  • git 配置：先 git --version 验证是否已装；没装时 macOS 通常会弹框提示安装 Command Line Tools，也可手动运行 xcode-select --install。
  • GitHub 认证推荐路径：brew install gh → gh auth login（选 HTTPS → Yes → 浏览器授权）；一次配好，自动存好 git 凭据，之后 git push 免密码。
  • GitHub Pages 开启：建仓库 physics-notes（Public）→ push → Settings → Pages → Deploy from a branch → main → /(root) → Save；约 10 分钟后访问 https://<用户名>.github.io/physics-notes/。
  • 主线示例①完成：课堂笔记 → 网页（HTML）→ 发布到 GitHub Pages，随时随地复习，一条链接分享给任何人。

一句话回顾：Markdown 让 AI 生成内容、HTML 让浏览器显示、git push 让 GitHub Pages 发布上线——把这三步串起来，你的笔记就从本地草稿变成了随时随地可访问的网页。

6 · 第二大脑

为什么要学这个：Ch5 你把 Physics 笔记变成了一个网页发到了 GitHub Pages——这是产出的第一步：把它发出去。但还有另一个方向：把学到的东西留下来，积累成真正属于你的知识库，下次用得上。这就是「第二大脑（second brain）」的核心想法：在大脑之外建一个外部数字库，把你读过、学过、想过的东西系统地存进去。以前靠人工整理，现在 AI 可以帮你维护——这一章就把这件事做起来。

「第二大脑」是谁提出的

「第二大脑（second brain）」这个叫法来自 Tiago Forte 于 2017 年推出的 BASB（Building a Second Brain）课程，2022 年出版成书。他在官网给出的核心定义是：

"A Second Brain is an external, centralized, digital repository for the things you learn and the resources from which they come."

翻译过来就是：一个外挂在大脑之外、集中存放你所学一切的数字仓库。Forte 的框架叫 C.O.D.E.：Capture（捕捉）/ Organize（组织）/ Distill（提炼）/ Express（输出）。

AI 原生版本：Karpathy 的「LLM Wiki」

2026 年 4 月，AI 研究员 Andrej Karpathy 在 X（Twitter）上描述了一种用 LLM 构建个人知识库的思路，随后在 GitHub 发布了一份可直接复制给 agent 使用的「idea file」（Gist）。他本人从未在原文中使用「second brain」这个词——他管这个东西叫 LLM wiki 和 "personal knowledge base"。

Karpathy Gist 里有一句话最能点清这种模式的本质：

"Obsidian is the IDE; the LLM is the programmer; the wiki is the codebase."

他还说："The wiki is a persistent, compounding artifact"——知识库是持续增长的制品，不是每次用完就扔的东西。这和传统的 RAG（每次查询都去重新检索原始文档）不一样：wiki 由 LLM 维护，知识被编译进去一次，下次直接用，越积越厚。

值得一提的是，他的精神前身指的是 Vannevar Bush 在 1945 年构想的 Memex——一台帮人管理个人资料的机器。Karpathy 说："The part he couldn't solve was who does the maintenance"——而现在，LLM 解决了这个维护难题。

⚠️ 归因说明

「第二大脑 / Building a Second Brain（BASB）」是 Tiago Forte 的方法论与品牌，与 Karpathy 无关。社区在二次传播时常把 Karpathy 的 LLM wiki 贴上「second brain」标签，但那是社区的叫法，不是他本人的措辞。本章用「第二大脑」作为熟悉的伞状标题，但归属按事实说清楚。（Karpathy 原推因 X.com 需登录无法直接核实全文，原话引述来自经核实的 Gist 原文及可信二次来源。）

最小形态：一个 ~/brain/ 文件夹就够了

别被「知识库」这三个字吓到。在技术上，第二大脑最小的形态就是：

  • 你磁盘上的一个文件夹，比如 ~/brain/
  • 里面全是 .md 文件（Markdown 格式，纯文本，第 5 章你已经很熟了）
  • Claude Code 能直接读写这个文件夹——不需要安装任何额外的东西

就这样。你现在就能开始——不需要下载任何 app，不需要注册任何账号，不需要配置任何东西。先把库建起来、把 agent 用起来，其他的都可以之后再说。

Obsidian：可选的增强层（不装也行）

Obsidian（本地知识管理应用）是一个可以让这个文件夹变得更好用的 app——但它完全是可选的。

Obsidian 的关键事实：

  • 官网原文："Obsidian stores your notes locally as plain text Markdown files"——它不改变你的文件格式，vault（库）就是磁盘上的一个普通文件夹。
  • 你在 ~/brain/ 上打开 Obsidian，就能多出双向链接（bidirectional links）、图谱视图（graph view）和 1000+ 社区插件——但底层的 .md 文件完全不变，Claude Code 照样能读写。
  • 官网原文："you're never locked in. You own your data for the long term." — 没有供应商锁定，文件永远是你的。
  • 个人使用核心功能完全免费；可选付费同步。
  • 支持 macOS / Windows / Linux / iOS / Android。

一句话总结 Obsidian 的定位：Claude Code 是写手，Obsidian 是书架上的导航系统——写手能独立工作，但有导航系统更好找书。装不装，取决于你是否需要可视化那张笔记网络图。不装的话，~/brain/ 文件夹照样运转。

示例②③ 在这里汇流

前几章你一直在跑两条线：

  • 示例①：Physics 笔记 → HTML → GitHub Pages（Ch5 完成）
  • 示例②：第二大脑——用 Obsidian + Claude 共建知识库（本章起步）
  • 示例③：机械工程（mechanical engineering）兴趣拓展

从这一章开始，示例② 和 ③ 在同一个库里汇流：

  • Physics 笔记沉淀进 ~/brain/physics/——和 Ch5 发出去的网页互为补充，一个公开展示、一个私下积累
  • 机械工程兴趣探索的笔记进 ~/brain/mechanical/
  • 两个文件夹都住在同一个 ~/brain/ 库里，Claude Code 可以跨文件夹检索、整合、建立联系

以后你每次学到新东西——不管是 Physics 课、还是看了一个关于涡轮发动机的视频——都可以让 agent 帮你整理、沉淀进对应的文件夹。知识库会越来越厚，而维护的成本几乎为零。

🙌 跟我做 · 建库，沉淀第一条笔记

第一步：建立 ~/brain/ 及子文件夹

在 terminal 里运行：

mkdir -p ~/brain/physics ~/brain/mechanical

这一行就把整个库架构建好了：~/brain/ 是根目录，两个子文件夹分别对应两条兴趣线。

第二步：进入库，启动 Claude Code

cd ~/brain
claude

第三步：让 agent 把一条 Physics 要点沉淀进来

在 Claude Code 对话里输入（context 交代清楚）：

我在学 Physics（O-level），刚学完「牛顿三大运动定律」这个主题。
请帮我在 physics/newton-laws.md 里写一份简洁的知识点总结：
- 三条定律各用一句中文写清楚核心
- 每条附一个生活化例子
- 文件顶部加一行 # 标题和今天的日期

Claude Code 会创建 ~/brain/physics/newton-laws.md 并写入内容。它动手前会告诉你计划，确认没问题就让它继续。

第四步：试试机械工程方向

接着在同一个对话里输入：

我对机械工程感兴趣，特别是内燃机（internal combustion engine）的工作原理。
请帮我在 mechanical/internal-combustion-engine.md 里写一份入门笔记：
- 四冲程循环（进气、压缩、做功、排气）各一句话说清楚
- 和 Physics 里「功和能」的联系点一下
- 文件顶部加标题和日期

看到 ~/brain/mechanical/internal-combustion-engine.md 被创建，示例②③就在这一刻在同一个库里汇流了。

第五步：验收 + 检索

在对话里问 agent：

帮我列出 brain 库里目前有哪些笔记，每个文件一行，写出文件路径和首行标题

它会用 find 或 ls 帮你扫一遍库，返回索引。这就是 Karpathy 说的「检索」——不是问 AI 一个问题然后忘掉，而是去你自己积累的知识库里找已经整理好的答案。

🧠 心法回扣 · 持久制品 vs. 一次性对话

Ch0 说的「上下文→产出→eval→修正」闭环，在这里有了新维度：产出不只是一个回答，而是一份持久留在磁盘上的笔记。每一次对话都在往库里添砖——Karpathy 的原话是「compounding artifact（复利制品）」。对话窗口关了，知识还在。下次你问任何关于内燃机的问题，可以先说「去看我 ~/brain/mechanical/ 里的笔记」，给 agent 上下文，得到的答案会更贴合你已有的知识框架。

这也是为什么 Obsidian 的那条类比这么准：Obsidian 是 IDE，LLM 是程序员，wiki 是代码库。你维护代码库，程序员往里写代码，IDE 帮你看清结构——三者各司其职。

小结

  • 「第二大脑」这个叫法来自 Tiago Forte（BASB）；Karpathy 提出的是「LLM wiki」模式——两者理念相近，归属不同，本章并列介绍。
  • 最小形态就是 ~/brain/ 一个 Markdown 文件夹——Claude Code 能直接读写，不需要安装任何东西就能开始。
  • Obsidian 是可选的增强层，在同一个文件夹上加双链、图谱和插件，底层文件格式不变，可选，不装也行。
  • 示例②（第二大脑）和③（机械工程兴趣）在 ~/brain/ 汇流：Physics 进 physics/、机械工程进 mechanical/，Claude Code 跨文件夹管理整个库。

一句话回顾：第二大脑的本质是一个 ~/brain/ 文件夹——AI 当维护员，你负责提供素材和审阅，知识越积越厚，不会因为对话窗口关掉而消失。

7 · 扩展能力：MCP 与 Skills

为什么要学这个：到现在，你用的是 Claude Code 自带的那一套工具——读文件、改文件、跑命令。这已经很强了。但假如你想让它读你的 Google 日历、连上某个数据库、或者把你常用的一套整理流程封装成一个命令随时调——那就需要"插件"了。这一章介绍两种扩展机制：MCP（接外部系统的标准插头）和Skills（把专长/流程打包成工具）。这章故意做得很轻——你不需要现在就用，只要知道"这东西存在、需要时可以加"就够了。

MCP：AI 的 USB-C 接口

MCP（Model Context Protocol，模型上下文协议）是一套开放标准，解决的是「AI 怎么统一连上外部工具和数据」这个问题。官方的定义是："an open-source standard for connecting AI applications to external systems"（连接 AI 应用与外部系统的开源标准）。

官方给的比喻非常直白：MCP 就是 AI 的 USB-C 接口。就像 USB-C 统一了电子设备的连接规格，MCP 统一了「AI 插上外部系统」这件事的规格——不管对面是文件系统、数据库、搜索引擎还是你的 Google Calendar，都用同一套接法。Claude、ChatGPT、VSCode Copilot 等工具都支持这个标准，所以插一次、到处能用。

对你来说，MCP 的实际意义是：当你想让 Claude Code 做一件它自带工具做不到的事（比如直接读你的 Notion 页面），通常就是去找有没有对应的 MCP server 装上。

Skills：把专长打包成一个命令

Skills（技能）是把一套重复用的指令、流程或清单打包成一个随时可调的「专长」。官方说法是："Skills extend what Claude can do. Create a SKILL.md file with instructions, and Claude adds it to its toolkit."（Skills 扩展 Claude 能做的事——写一个 SKILL.md 文件，Claude 就把它加进工具箱。）

什么时候该建一个 Skill？官方给了个很实在的判断标准：当你发现自己每次对话都在重复粘贴同一套指令，或者 CLAUDE.md 里某段「背景说明」已经长成了一套操作流程——那就该把它封装成 Skill 了。封装完，你可以直接用 /技能名 调它，或者让 Claude 在遇到对应情境时自动调用。Skills 遵循开放的 Agent Skills 标准，跨工具通用。

顺便认识：Subagents 与 Hooks

Subagents（子智能体）是在独立的上下文窗口里跑单项任务、只把结果汇报给主对话的专属助手——官方描述："specialized AI assistants that handle specific types of tasks"，适合把「查资料、搜日志」这类会淹没主对话的侧任务交出去处理。

Hooks（钩子）是你写好、在 Claude Code 运行的特定节点自动触发的脚本——官方定义："user-defined shell commands, HTTP endpoints, or LLM prompts that execute automatically at specific points in Claude Code's lifecycle"。典型用法：每次改完文件自动跑格式检查，或者拦截危险命令。

🧠 理念：按需添加，别一上来堆一堆

MCP、Skills、Subagents、Hooks——这些都是扩展层，不是必须一开始就配齐的。实习生能把活干完，你再给他多一件工具；工具还没用到就先搬一堆进来，反而乱。遇到自带工具做不到的事，再去找对应的扩展——这才是正确的节奏。现在知道「这东西存在」就够了，需要时翻官方文档加上去就行。

小结

  • MCP：AI 的 USB-C 接口——统一规格，让 Claude 连上任意外部系统（文件、数据库、日历……）。
  • Skills：把重复用的指令/流程打包成一个可随时调用的「专长」，放进 .claude/skills/。
  • Subagents：专门处理侧任务的子智能体，跑在独立上下文里，只汇报结果。
  • Hooks：在 Claude Code 的特定节点自动触发的脚本，用来自动化和把关。

一句话回顾：这四件东西都是「需要时插上去的扩展」——现在不用急着学怎么配，知道它们存在、遇到瓶颈时知道往哪找，就是这章的全部目标。
